# Deployment

Production layout: frontend on **GitHub Pages** behind **flusk.org**, backend on **Render** (Docker), database on **Neon** (Postgres), live state on **Render Key Value** (Redis-compatible).

```
flusk.org  ‚îÄ‚îÄ‚ñ∫  GitHub Pages (Angular SPA)
                    ‚îÇ HTTPS + WSS
                    ‚ñº
fluks-api.onrender.com  ‚îÄ‚îÄ‚ñ∫  Neon Postgres (durable data)
                        ‚îî‚îÄ‚îÄ‚ñ∫  Render Key Value (live room state)
```

## 1. Database ‚Äî Neon

1. Create a project at [neon.tech](https://neon.tech) (free tier).
2. Copy the connection string and append the schema parameter:
   ```
   postgresql://USER:PASSWORD@HOST/neondb?sslmode=require&schema=public
   ```
3. Nothing else ‚Äî migrations run automatically when the backend container starts (`prisma migrate deploy`).

> Why not Render's free Postgres: it expires 30 days after creation (then gets suspended and deleted), and the free tier allows only one database per workspace. Neon's free tier is permanent and lives outside Render's limits.
>
> **Temporary alternative ‚Äî reuse an existing Render Postgres:** if a free Postgres from a previous deployment still exists in your workspace, you can point `DATABASE_URL` at it (Dashboard ‚Üí the database ‚Üí Info ‚Üí *Internal Database URL* if the API runs in the same workspace/region, otherwise *External*). The schema migrations are identical, so `prisma migrate deploy` is a no-op and existing data survives. Check its **expiry date** on the database page ‚Äî plan the move to Neon before that day.

## Render free-tier limits (one workspace)

- **1 free Postgres** ‚Äî expires 30 days after creation.
- **1 free Key Value (Redis)** ‚Äî if an old instance occupies the slot, delete it (Dashboard ‚Üí old instance ‚Üí Settings ‚Üí Delete) before applying the blueprint; the old Vejas deployment is obsolete anyway. Same for the old web service ‚Äî free web services are not limited to one, but keeping both wastes nothing except clarity.

## 2. Backend ‚Äî Render Blueprint

1. Render dashboard ‚Üí **New ‚Üí Blueprint** ‚Üí select the `fluks` GitHub repository.
2. Render reads [render.yaml](../render.yaml) and creates:
   - `fluks-api` ‚Äî Docker web service built from `backend/` (free plan);
   - `fluks-redis` ‚Äî Key Value instance (free plan), `REDIS_URL` wired automatically.
3. When prompted, paste `DATABASE_URL` (the Neon string from step 1). JWT secrets are auto-generated.
4. After the first deploy, verify: `https://fluks-api.onrender.com/health`.
5. **Check the actual service URL** on the service page. If the global name `fluks-api` was taken, Render appends a suffix (e.g. `fluks-api-x3k2.onrender.com`) ‚Äî then update `frontend/src/environments/environment.prod.ts` (`apiUrl` and `socketUrl`) and push.

Environment (managed by the blueprint):

| Variable | Source |
|---|---|
| `DATABASE_URL` | entered manually (Neon) |
| `REDIS_URL` | from the Key Value instance |
| `JWT_SECRET_KEY`, `JWT_SECRET_REFRESH_KEY` | generated by Render |
| `CORS_ORIGIN` | `https://flusk.org,https://www.flusk.org` |
| `CRYPT_SALT`, `TOKEN_EXPIRE_TIME`, `TOKEN_REFRESH_EXPIRE_TIME` | fixed values |

> Free-plan note: the service spins down after ~15 min idle; the first request afterwards takes ~50 s.

## 3. Frontend ‚Äî GitHub Pages

1. Repository ‚Üí **Settings ‚Üí Pages** ‚Üí Source: **GitHub Actions**.
2. Push to `main` (or run the `Deploy frontend` workflow manually) ‚Äî it builds with `--base-href /`, adds `CNAME` and a `404.html` SPA fallback, and publishes.
3. After DNS is set up (step 4): Settings ‚Üí Pages ‚Üí **Custom domain** = `flusk.org` ‚Üí wait for the check ‚Üí enable **Enforce HTTPS**.

## 4. DNS ‚Äî flusk.org

|---|---|---|
| A | `@` | `185.199.108.153` |
| A | `@` | `185.199.109.153` |
| A | `@` | `185.199.110.153` |
| A | `@` | `185.199.111.153` |

Propagation is usually minutes; the Pages TLS certificate is issued automatically after the domain check passes (typically 10‚Äì30 min).

## 5. Smoke test

1. Open `https://flusk.org` ‚Äî home page loads, no console errors.
2. Sign up, log in.
3. Create a room, open it in a second browser/incognito with another account.
4. Play/pause/seek ‚Äî playback syncs; send a chat message ‚Äî it appears for both.
5. Deep-link check: open `https://flusk.org/rooms` directly (404 fallback must serve the SPA).

## CI/CD summary

| Workflow | Trigger | What it does |
|---|---|---|
| `ci.yml` | push/PR to `main`/`develop` | frontend: lint + Vitest; backend: Jest (path-filtered) |
| `deploy-frontend.yml` | push to `main` touching `frontend/**` | build + publish to GitHub Pages |
| Render autodeploy | push to `main` touching `backend/**` | rebuild Docker image, run migrations, restart |
