# Deployment

Production layout: frontend on **GitHub Pages** behind **fluks.org**, backend on **Render** (Docker), database on **Neon** (Postgres), live state on **Render Key Value** (Redis-compatible).

```
fluks.org  ──►  GitHub Pages (Angular SPA)
                    │ HTTPS + WSS
                    ▼
fluks-api.onrender.com  ──►  Neon Postgres (durable data)
                        └──►  Render Key Value (live room state)
```

## 1. Database — Neon

1. Create a project at [neon.tech](https://neon.tech) (free tier).
2. Copy the connection string and append the schema parameter:
   ```
   postgresql://USER:PASSWORD@HOST/neondb?sslmode=require&schema=public
   ```
3. Nothing else — migrations run automatically when the backend container starts (`prisma migrate deploy`).

> Why not Render's free Postgres: it expires after 30 days; Neon's free tier is permanent.

## 2. Backend — Render Blueprint

1. Render dashboard → **New → Blueprint** → select the `fluks` GitHub repository.
2. Render reads [render.yaml](../render.yaml) and creates:
   - `fluks-api` — Docker web service built from `backend/` (free plan);
   - `fluks-redis` — Key Value instance (free plan), `REDIS_URL` wired automatically.
3. When prompted, paste `DATABASE_URL` (the Neon string from step 1). JWT secrets are auto-generated.
4. After the first deploy, verify: `https://fluks-api.onrender.com/health`.

Environment (managed by the blueprint):

| Variable | Source |
|---|---|
| `DATABASE_URL` | entered manually (Neon) |
| `REDIS_URL` | from the Key Value instance |
| `JWT_SECRET_KEY`, `JWT_SECRET_REFRESH_KEY` | generated by Render |
| `CORS_ORIGIN` | `https://fluks.org,https://www.fluks.org` |
| `CRYPT_SALT`, `TOKEN_EXPIRE_TIME`, `TOKEN_REFRESH_EXPIRE_TIME` | fixed values |

> Free-plan note: the service spins down after ~15 min idle; the first request afterwards takes ~50 s.

## 3. Frontend — GitHub Pages

1. Repository → **Settings → Pages** → Source: **GitHub Actions**.
2. Push to `main` (or run the `Deploy frontend` workflow manually) — it builds with `--base-href /`, adds `CNAME` and a `404.html` SPA fallback, and publishes.
3. After DNS is set up (step 4): Settings → Pages → **Custom domain** = `fluks.org` → wait for the check → enable **Enforce HTTPS**.

## 4. DNS — fluks.org

At the domain registrar:

| Type | Host | Value |
|---|---|---|
| A | `@` | `185.199.108.153` |
| A | `@` | `185.199.109.153` |
| A | `@` | `185.199.110.153` |
| A | `@` | `185.199.111.153` |
| CNAME | `www` | `<github-username>.github.io` |

Propagation is usually minutes; the Pages TLS certificate is issued automatically after the domain check passes (typically 10–30 min).

## 5. Smoke test

1. Open `https://fluks.org` — home page loads, no console errors.
2. Sign up, log in.
3. Create a room, open it in a second browser/incognito with another account.
4. Play/pause/seek — playback syncs; send a chat message — it appears for both.
5. Deep-link check: open `https://fluks.org/rooms` directly (404 fallback must serve the SPA).

## CI/CD summary

| Workflow | Trigger | What it does |
|---|---|---|
| `ci.yml` | push/PR to `main`/`develop` | frontend: lint + Vitest; backend: Jest (path-filtered) |
| `deploy-frontend.yml` | push to `main` touching `frontend/**` | build + publish to GitHub Pages |
| Render autodeploy | push to `main` touching `backend/**` | rebuild Docker image, run migrations, restart |
